# 当前架构总览

这篇文档是 `02-架构重构` 目录中唯一需要优先阅读的文档。

目的只有一个：

- 让新人用最短时间理解项目现在的真实结构
- 避免被旧方案、旧计划、旧术语带偏

如果你只看一篇架构文档，就看这一篇。

## 1. 当前架构一句话

项目当前采用的是：

**装配层 + 核心大组件 + 弱 Adapter 增强层**

也就是：

- 默认逻辑仍然保留在 `TocMindmap.ts`
- Typora 专属能力通过 `TyporaAdapter` 提供增强
- 缺少强 adapter 能力时，核心组件仍然可以用 fallback 跑基础版

这不是“强平台适配架构”，也不是“完全写死 Typora”。

它是一种偏务实的折中：

- 保住当前 Typora 体验
- 不把未来迁移到 VSCode / Obsidian 的路堵死
- 不为了抽象而把整个项目拆得过头

## 2. 代码分层

### 2.1 装配层

文件：

- `src/main.ts`

职责：

- 加载插件设置
- 初始化 markmap 资源
- 创建 `TocMindmapComponent`
- 创建 `FloatingButtonComponent`
- 监听设置变化并把新配置下发给组件
- 提供 Typora 专属增强实现 `TyporaAdapter`

你可以把它理解成项目的“胶水层”。

### 2.2 核心组件层

文件：

- `src/components/TocMindmap.ts`
- `src/components/FloatingButton.ts`

职责：

- `TocMindmap.ts`
  负责 Markdown 预处理、markmap 渲染、节点点击跳转、当前标题定位、fit/highlight、实时更新、窗口交互等核心流程
- `FloatingButton.ts`
  负责悬浮按钮渲染、拖拽、图标更新

注意：

`TocMindmap.ts` 仍然是当前项目最重的核心文件。  
它还没有被彻底拆成“纯 core / 纯 UI / 纯 platform integration”三层。

### 2.3 平台增强层

文件：

- `src/main.ts` 中的 `TyporaAdapter`

职责：

- 提供 Typora 的稳定标题引用
- 用 `cid` / `nodeMap.toc.headers` / `#write` 做标题定位增强
- 提供更稳的滚动跳转和当前标题判断

这里的关键点不是“把逻辑都搬进 adapter”，而是：

**adapter 只负责增强，不是系统唯一前提。**

## 3. 当前 Adapter 设计

当前接口定义在：

- `src/components/TocMindmap.ts`

核心接口是 `IEditorAdapter`。

设计原则是：

- 基础能力尽量少
- 增强能力全部可选

当前关键能力分两类。

### 3.1 基础能力

- `getMarkdown()`
- `resolveImagePath()`

这两项是核心流程真正离不开的能力。

### 3.2 增强能力

- `getHeadingRefs?()`
- `resolveHeadingElement?()`
- `scrollToHeading?()`
- `getCurrentVisibleHeadingId?()`

这些能力有就优先用，没有就回退到组件内部默认逻辑。

这就是当前架构最重要的设计选择：

> 默认逻辑在 `TocMindmap.ts`，`TyporaAdapter` 只覆盖 Typora 独有优化。

## 4. 为什么要做成“弱 Adapter”

原因很直接：

- 如果走“强 Adapter 中心化”，Typora 结构会更干净
- 但以后迁移到 VSCode / Obsidian，往往就得先重写一整套 adapter
- 这会损失当前项目原本那种“整块复用 `TocMindmap.ts`，只修少量平台差异”的优势

所以当前方案故意保留了 fallback：

- 在 Typora 中，优先走更稳的增强链路
- 在其他平台中，就算 adapter 没完全实现，也还能跑一个基础版

这是一种偏迁移友好的设计。

## 5. 当前标题跳转链路

这是最近几轮改动里最关键的部分。

### 5.1 现在的做法

当前跳转不是只靠 DOM 标题标签，也不是只靠 `cid`。

而是分两层：

1. 优先走稳定 ID / Typora 增强链路
2. 失败后回退到文本匹配 / DOM fallback

### 5.2 Typora 增强链路

大致流程：

1. `TyporaAdapter.getHeadingRefs()` 提供 `HeadingRef[]`
2. `HeadingRef.id` 在 Typora 下通常就是 `cid`
3. `TocMindmap.ts` 在 `_syncMapsAfterRender()` 中建立
   - `state.path -> headingId`
   - `state.path -> HTMLElement`
   - `HTMLElement -> state.path`
4. 点击节点时优先用 `headingId` 反查标题块
5. 滚动高亮时优先用当前可见 `headingId`

### 5.3 Fallback 链路

如果平台没有提供增强能力，组件内部仍会尝试：

- 扫描 DOM 标题生成 fallback `HeadingRef`
- 通过文本内容匹配标题
- 通过普通 `scrollIntoView` 完成跳转

这就是当前“能增强，但不锁死”的核心实现。

## 6. 为什么 `cid` 重要，但不能污染核心架构

`cid` 对 Typora 很重要，因为它更接近 Typora 自己的块模型。

它解决的问题是：

- 标题编辑态下 DOM 结构会变
- 但块身份往往比 DOM 结构更稳定

所以在 Typora 环境里，`cid` 很有价值。

但 `cid` 不是 Markdown 生态通用能力，因此不能把核心架构写成：

- 没有 `cid` 就跑不了

正确做法就是现在这种：

- core 只认抽象的 `headingId`
- Typora 上由 `cid` 提供这个 `headingId`
- 其他平台以后可用自己的 block id / range / position 替代

## 7. 当前最重要的状态结构

`TocMindmap.ts` 里最值得理解的是这几张映射表：

- `headingElements: Map<string, HTMLElement>`
  从 markmap 节点 `state.path` 到标题元素
- `headingIds: Map<string, string>`
  从 markmap 节点 `state.path` 到稳定标题 ID
- `elementToPath: Map<HTMLElement, string>`
  从标题元素到 markmap 节点 `state.path`

它们负责把两套系统接起来：

- Markmap 节点树
- 编辑器标题块

这里要特别注意：

`state.path` 仍然是导图节点世界里的关键索引。  
但“正文标题身份”现在优先靠 `headingId`，而不是单纯靠文本路径。

## 8. 当前哪些旧说法已经过时

下面这些说法，已经不能代表当前代码现状：

- “核心唯一标识是层级文本路径”
- “应该把所有平台逻辑都下沉到 Adapter”
- “组件应该彻底变成纯 UI”

当前真实情况是：

- 导图侧索引依然大量使用 `state.path`
- 标题身份侧优先使用 `headingId`
- 平台逻辑只下沉了一部分
- `TocMindmap.ts` 仍然是核心大组件

所以现在不要再用“纯 adapter 架构”来理解这个项目。
